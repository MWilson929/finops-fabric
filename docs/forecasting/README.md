# Azure Cost Forecasting Hub

Complete platform for building production-grade cost forecasting on Microsoft Fabric using FOCUS-formatted Azure billing data.

## 📦 Package Contents

### Notebooks (Jupyter)
- **focus_cost_forecast.ipynb** — Main forecasting pipeline (StatsForecast/MSTL+AutoARIMA, Polars, 365-day horizon)
- **focus_cost_synthesise.ipynb** — Data extension utility (synthesises 23 months of realistic historical data from 1-month sample)

### Documentation
- **forecasting_handover.docx** — Comprehensive handover guide with architecture, setup instructions, and next steps
- **forecasting_roadmap_v3.docx** — Complete 13-item roadmap across v2/v3/Future phases with rationale for each item
- **README.md** — This file

## 🚀 Quick Start

### Prerequisites
- Microsoft Fabric workspace with Lakehouse
- FOCUS-formatted Azure cost data (50M+ rows)
- Python runtime with: `statsforecast`, `polars`, `deltalake`

### 1. Setup (5 min)
1. Update workspace GUID in both notebooks (`<workspace-guid>` placeholders)
2. Verify table names: `focus_cost` (input), `Finops_Hub` (lakehouse name)
3. Confirm FOCUS columns exist: `ChargePeriodStart`, `BilledCost`, `SubAccountId`

### 2. Test with Synthetic Data (10 min)
```
Run: focus_cost_synthesise.ipynb → creates 24 months of history from 1-month sample
```

### 3. Run Forecast (15 min)
```
Run: focus_cost_forecast.ipynb → produces daily actuals + 365-day forecast
Output tables: focus_actuals_daily, focus_forecast_daily
```

### 4. Connect to Power BI
- Create semantic model pointing to `vw_focus_actuals_and_forecast` (SQL view auto-generated by notebook)
- Build dashboard with actuals vs. forecast visualization

## 🎯 Architecture at a Glance

**Input**: FOCUS billing data (50M+ rows)
↓
**Processing**: Polars lazy aggregation to daily totals per subscription
↓
**Forecasting**: StatsForecast MSTL+AutoARIMA (single fit() call across all series)
↓
**Output**: Delta tables (actuals + forecast) + SQL view for Power BI
↓
**Consumption**: Power BI dashboard with ribbon/slicer controls

## 📋 Why This Stack?

| Component | Why |
|-----------|-----|
| **Polars** (not Spark) | Faster on 50M rows, lazy evaluation, streaming collect, single-node efficient |
| **StatsForecast** (not Prophet) | Polars-native, automatic per-series model selection, handles multi-seasonality well |
| **Daily granularity** | Matches FOCUS structure, right level for cost reporting |
| **Separate actuals/forecast tables** | Independent update cycles, cleaner Power BI design |
| **SQL view for Power BI** | Union in SQL (efficient) rather than Power BI DAX |

## 💾 Storage & Maintenance

- **Actuals table**: Appends only (immutable history)
- **Forecast table**: Overwrite on each run (365-day rolling window)
- **Sync cadence**: Daily recommended (after daily cost ingestion)
- **Retention**: Keep last 12 months of forecasts for accuracy tracking (for backtesting)

## 🔧 Customization Points

In `focus_cost_forecast.ipynb`:
- **DATE_COL**, **COST_COL**: Adjust to your FOCUS column names
- **HORIZON_DAYS**: Change forecast window (default 365)
- **FY_START_MONTH**: Fiscal year definition (default 9 = September)
- **vCores**: Bump to 16 for >100M rows

## 📊 Key Outputs

**focus_actuals_daily**
```
ds (date) | unique_id (subscription) | y (cost)
```

**focus_forecast_daily**
```
ds | unique_id | yhat (point forecast) | model_id | forecast_generated_at
```

**vw_focus_actuals_and_forecast** (unified SQL view)
```
date | subscription_id | cost | record_type (actual/forecast) | model_id | forecast_generated_at
```

## ⚠️ Known Limitations

- **No prediction intervals yet** (coming in v2 — quantile outputs)
- **Single model ensemble**: Currently MSTL+AutoARIMA only (ensembling planned for v3)
- **No exogenous features**: Calendar/deployment events not yet integrated (roadmap item)
- **Cold-start**: New subscriptions need manual policy decision (roadmap item 12)

## 🚨 Troubleshooting

**ABFSS path errors**
→ Verify workspace GUID in notebook parameters

**OutOfMemory on 50M rows**
→ Bump vCores to 16 in %%configure cell

**Missing ChargePeriodStart column**
→ Verify FOCUS table structure; adjust DATE_COL parameter

**Forecast seems high/low**
→ Check COST_COL parameter (BilledCost vs EffectiveCost affects baseline)
